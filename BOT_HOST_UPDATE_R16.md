# Seller.pro.uz r16 — сверка отчётов и документы по потерям

Версия: `2026.07.22-premium-r16-report-reconciliation`

## Перед обновлением

1. В Telegram выполните `/backup_db` от имени владельца.
2. Сохраните текущие переменные окружения BotHost.
3. Не удаляйте `bot.db`, `.env` и каталог с пользовательскими данными.

## Установка на BotHost

Это обновление состоит не только из `main.py`. Загрузите с заменой все файлы из архива:

- `main.py`
- `market_plus_reports.py`
- `formatters.py`
- `uzum_client.py`
- `requirements.txt`

После загрузки выполните полную пересборку/перезапуск, чтобы BotHost установил новую зависимость `python-docx` из `requirements.txt`.

В журнале запуска должны появиться строки:

```text
PREMIUM_RELEASE_LOADED version=2026.07.22-premium-r16-report-reconciliation
REPORT_RECONCILIATION_LOADED: historical purchasePrice + issued-date + cancelled quantity + tax/ROI
FBO_ACCEPTANCE_RECONCILIATION_LOADED: invoice/product totals + stale-zero guard
MARKET_STYLE_REPORTS_LOADED: daily PDF/Excel + loss/damage claim documents
```

Если появляется `ModuleNotFoundError: No module named 'docx'`, BotHost не переустановил зависимости. Запустите повторную сборку приложения, а не только обычный рестарт процесса.

## Новые команды и разделы

- Отчёт за сегодня: `📊 Отчёты → 📋 Дневной отчёт` или `/market_report today`.
- Отчёт за вчера: `/market_report yesterday`.
- За конкретный день: `/market_report 2026-07-21`.
- Документы по потерям/повреждениям: `📦 Остатки → 📑 Документы по потерям` или `/loss_documents`.
- Выбор налога: `/finance_settings → 🏛 Выбрать налог`.

Доступные быстрые ставки: без налога, налог с оборота 1%, НДС 6%, НДС 12% и произвольная ставка.

## Как теперь считаются показатели

- Себестоимость берётся из исторического `purchasePrice` каждой финансовой строки Uzum. Текущий `purchasePrice` каталога используется только как резервный источник.
- Налог считается с выручки товара.
- Чистая прибыль товара: выплата Uzum минус себестоимость минус налог.
- ROI: чистая прибыль товара / себестоимость.
- Ориентир выплаты за текущую и предыдущую дату строится по выданным заказам (`dateIssued`) и сумме `sellerProfit`; это расчётный показатель, а не подтверждение банковского зачисления.
- Комиссия и логистика показываются в расходах, но не вычитаются второй раз, если они уже учтены в выплате Uzum.
- Если части `purchasePrice` или расходов нет, бот пишет «Результат по известным данным» и показывает процент покрытия себестоимостью.
- Размер возмещения в претензиях рассчитывается только при наличии точной текущей цены и точной комиссии SKU из Uzum. Неполные строки оставляются без выдуманного значения.

## Быстрая проверка после запуска

1. Откройте `/finance_settings`, выберите нужную ставку налога.
2. Сформируйте `/market_report yesterday` и откройте оба файла — PDF и Excel.
3. Откройте `/loss_documents` и убедитесь, что Excel/Word сформированы для найденных потерь и повреждений.
4. Проверьте завершённую FBO-поставку: итог в уведомлении должен совпадать с деталями товаров накладной.
5. Дождитесь новой отмены: сообщение должно содержать схему и причину, если Uzum передал эти поля.
