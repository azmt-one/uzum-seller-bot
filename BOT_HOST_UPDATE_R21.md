# Seller.pro.uz r21 — названия товаров Uzum и защита фоновых задач

Версия: `2026.07.22-premium-r21-product-title-stability`

## Что исправлено

- Во всех клиентских отчётах основным названием стало документированное поле Uzum `productTitle` — название карточки товара.
- `skuTitle`, SKU ID и штрихкод больше не подставляются вместо названия товара.
- Если Uzum не передал `productTitle`, бот показывает `—`, а не вводит продавца в заблуждение техническим SKU.
- Одинаковые `productTitle` и `skuTitle` больше не выводятся дважды.
- Реально отличающийся размер, цвет или вариант может показываться отдельной строкой `Вариант / SKU`, но не склеивается с названием товара.
- Варианты одной карточки объединяются по товару в топе, остатках, дневных PDF/Excel и юнит-экономике. Количество, выручка и точная себестоимость каждого варианта при этом суммируются без потери данных.
- Названия исправлены в уведомлениях о продажах, отменах, остатках, потерях, FBO-приёмке, накладных, отзывах, дневных отчётах, полном PDF и Excel.

## Проверка данных Uzum

- Контракт сверён с актуальной схемой Uzum Seller OpenAPI:

  - `productTitle` — название продукта;
  - `skuTitle` — название SKU;
  - `date` и `dateIssued` — Unix Epoch в миллисекундах;
  - `purchasePrice` — закупочная цена;
  - `sellerProfit` — выплата продавцу;
  - `commission`, `logisticDeliveryFee`, `amountReturns`, `cancelled` обрабатываются отдельно.

- Finance API проверяется в документированном миллисекундном режиме и в совместимом режиме старых развёртываний Uzum.
- Бот проверяет даты реально возвращённых строк. Если Uzum неверно понял единицы времени и вернул чужой период, такой ответ отклоняется, а завышенный отчёт не формируется.
- Все страницы Finance и расходов загружаются с дедупликацией и ограничителями.
- `403 Shop is not available` и неактивный токен не превращаются в «0 продаж» или «0 расходов».
- Возвращённое списание Uzum (`REFUNDED`) корректно сторнирует исходный расход.

## Стабильность для действующих пользователей

- После `403` недоступный магазин ставится на часовую паузу отдельно для продаж, заказов, остатков, потерь и FBO-приёмки.
- Снимки продаж и остатков при ошибке API сохраняются и не заменяются пустыми данными.
- В журнал пишется одно понятное предупреждение без повторяющихся traceback.
- Если пользователь заблокировал бота, его автоотчёты, дайджесты и фоновые запросы приостанавливаются.
- После нового входящего сообщения пользователя доставка автоматически возобновляется.
- Для этого автоматически создаётся служебная таблица `telegram_delivery_state`. Существующие пользователи, магазины, токены и настройки не удаляются.

## Перед обновлением

1. Выполните `/backup_db` от имени владельца.
2. Сохраните текущие переменные окружения BotHost.
3. Не удаляйте `bot.db`, `.env`, API-ключи, шрифты и пользовательские данные.

## Установка на BotHost

Загрузите с заменой все файлы из архива:

1. `main.py`
2. `seller_reports.py`
3. `seller_pdf_report.py`
4. `formatters.py`
5. `uzum_finance.py`
6. `uzum_client.py`
7. `requirements.txt`

После замены выполните полную пересборку и перезапуск приложения.

В журнале должны появиться строки:

```text
PREMIUM_RELEASE_LOADED version=2026.07.22-premium-r21-product-title-stability
PRODUCT_TITLE_R21_LOADED: Uzum productTitle canonical + distinct variant/SKU
DELIVERY_GUARD_R21_LOADED: blocked users paused + API failures never shown as zero
```

## Быстрая проверка после установки

1. Откройте `📊 Отчёты → 📅 Отчёт за день → Сегодня`.
2. В PDF и Excel должно быть название карточки как в Uzum; одинаковое название SKU рядом не повторяется.
3. Откройте `📦 Склад → 📦 Все остатки`: бот должен отправить PDF, сгруппированный по товарам.
4. Откройте `💰 Продажи → 🏆 Топ товаров`: размеры и цвета одной карточки должны быть объединены.
5. Проверьте товар с действительно отличающимся вариантом: вариант может быть указан отдельно, но название товара остаётся главным.
6. Для недоступного магазина допустима одна строка `access unavailable`; повторных traceback и ложного нулевого отчёта быть не должно.

## Откат

Верните комплект файлов r20 и перезапустите приложение. Дополнительная таблица `telegram_delivery_state` безопасна и не мешает r20; удалять её не требуется.
